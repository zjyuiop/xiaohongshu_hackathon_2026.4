# 当前项目 UI 设计文档

## 1. 项目定位

这是一个基于 HarmonyOS ArkUI 的“时间人格 / 多阶段人格对话”应用。

核心玩法是：

1. 选择一个人物，或创建一个自定义人物。
2. 系统根据人物经历拆出多个时间阶段的人格切片。
3. 用户选择 2 到 3 个阶段人格进入“会议 / 对话”。
4. 系统输出讨论过程、共识、分歧与行动建议。
5. 讨论结果沉淀为可回看、可分享、可导出的纪要卡片。

从 UI 视角看，这个项目不是单一聊天产品，而是一个包含以下几类能力的复合型应用：

- 人物库浏览
- 人物创建与导入
- 时间线人格选择
- 多角色会议对话
- 讨论纪要沉淀
- 分享与导出

## 2. 当前产品结构

当前主入口是 `entry/src/main/ets/pages/Index.ets`，首页为三大 Tab 工作台：

- `角色`
- `会议`
- `纪要`

项目现有页面如下：

| 页面 | 文件 | 作用 |
|---|---|---|
| 首页工作台 | `pages/Index.ets` | 三大 Tab 总入口 |
| 人物详情页 | `pages/ProfileDetail.ets` | 查看人物简介、时间线、阶段人格 |
| 创建角色页 | `pages/CreateRole.ets` | 手动输入 / 微信记录 / AI 对话三种创建入口 |
| 探索人物页 | `pages/ExplorePersonas.ets` | 搜索与分类浏览预设人物 |
| 参会人选择页 | `pages/MeetingParticipantPage.ets` | 勾选 2~3 个阶段人格 |
| 会议聊天页 | `pages/MeetingChatPage.ets` | 展示会议过程与共识结果 |
| 纪要列表页 | `pages/SummaryListPage.ets` | 查看全部纪要、收藏、标签分类 |
| 纪要详情页 | `pages/SummaryDetailPage.ets` | 预览纪要卡片、分享、导出、收藏、删除 |

## 3. 核心用户流程

### 3.1 预设人物流程

1. 首页进入 `角色` Tab。
2. 浏览人物卡片或进入 `探索人物页` 搜索人物。
3. 进入 `人物详情页` 查看人生时间线。
4. 在时间线中选择阶段人格。
5. 将选中人格加入会议。
6. 进入 `会议` 或直接开始会议。
7. 查看讨论结果。
8. 结果自动进入 `纪要` 模块。

### 3.2 自定义人物流程

1. 首页点击“创建角色”或“导入人物”。
2. 在 `CreateRole.ets` 中输入人物名和人物传记。
3. 后端解析 biography，生成 timeline nodes。
4. 后端继续构建多个 Persona。
5. 进入人物详情页或直接进入会议链路。

### 3.3 纪要沉淀流程

1. 会议完成后，`SummaryService` 自动保存纪要。
2. 首页 `纪要` Tab 展示记录摘要。
3. 可进入纪要列表按标签、收藏、搜索查看。
4. 可在纪要详情页进行：
   - 分享
   - 导出图片
   - 导出文案
   - 生成视频回放文案
   - 收藏
   - 删除

## 4. 当前页面职责拆解

### 4.1 首页 `Index.ets`

首页是整个产品的“工作台”，包含三块内容：

- `角色 Tab`
  - 展示创建入口
  - 展示导入入口
  - 展示探索入口
  - 展示当前已收集人物卡片列表
- `会议 Tab`
  - 展示当前已选阶段人格
  - 输入讨论话题
  - 进入参会人选择页
  - 开始会议
- `纪要 Tab`
  - 展示近期会议纪要
  - 快速进入详情、分享、导出

当前首页已经有比较明显的视觉风格：

- 浅色背景
- 大面积柔和光斑
- 卡片化布局
- 轻玻璃拟态
- 圆角较大
- mint / lavender 系颜色为主

### 4.2 人物详情页 `ProfileDetail.ets`

这是“内容理解页”，重点不是操作密度，而是帮助用户理解人物。

页面包含：

- 人物名称
- 人物 biography
- 已选人格数量 / 已加入会议数量
- 人生时间线
- 每个阶段的人格入口按钮

时间线节点支持折叠 / 展开，展开后包含：

- 阶段摘要
- 关键事件
- traits
- values
- source evidence

这页很适合后续重点做“时间感”和“人格切片感”。

### 4.3 创建角色页 `CreateRole.ets`

当前是三种创建方式的入口页：

- 手动输入
- 微信记录
- AI 对话

但真正已打通的主流程是：

- 手动输入人物名 + biography

另外两种目前更像“预留模式入口”，还没有完整链路。

所以 UI 设计上可以明确区分：

- 可用能力
- 即将开放能力

### 4.4 探索人物页 `ExplorePersonas.ets`

这是人物库检索页，包含：

- 搜索框
- 分类 chip
- 人物结果卡片
- 热度感知
- 匹配原因标签

更偏“内容发现”和“人物导航”。

### 4.5 参会人选择页 `MeetingParticipantPage.ets`

这是一个很明确的选择页：

- 加载某个人物的全部阶段人格
- 用户勾选阶段人格
- 至少 2 个才能进入会议

这页是“决策页”，UI 重点应突出：

- 当前已选数量
- 每个阶段人格差异
- 选择后的状态反馈

### 4.6 会议聊天页 `MeetingChatPage.ets`

当前是“结果型会议页”，不是实时逐字生成页。

页面结构包括：

- 会议主题
- 会议消息流
- 最终 summary title
- 最终 consensus

后端 `PersonaApi.ets` 其实已经预留了 `streamArena()` 的 SSE 能力，但当前页面仍然主要使用 `runArena()` 一次性拿结果。

这意味着后续 UI 可升级方向很明确：

- 从“生成后展示”升级成“实时流式会议”

### 4.7 纪要列表页 `SummaryListPage.ets`

这是一个内容管理页，能力包括：

- 全部纪要
- 收藏纪要
- 标签筛选
- 搜索
- 按时间分组

当前气质更像“内容档案馆”。

### 4.8 纪要详情页 `SummaryDetailPage.ets`

这是目前最完整、最接近成品质感的展示页之一。

能力很多：

- 纪要 Hero 区
- 模板切换
  - `classic`
  - `quote`
  - `timeline`
- 预览卡片
- 信息面板
- 回放准备区
- 导出
- 分享
- 收藏
- 删除

这页适合作为后续整体视觉语言的“设计母版”。

## 5. 当前数据与业务模型

项目中的核心数据定义在：

- `entry/src/main/ets/common/Models.ets`

设计 UI 时最需要理解的是以下对象：

- `PresetProfile`
  - 人物基础资料
- `TimelineNode`
  - 某一阶段的人生节点
- `PersonaSpec`
  - 从某阶段抽出来的人格角色
- `ArenaMessage`
  - 会议中的单条消息
- `ArenaSummary`
  - 会议总结
- `SummaryRecord`
  - 本地沉淀后的纪要记录

简单理解：

- `Profile` 解决“这个人是谁”
- `TimelineNode` 解决“这个人经历了什么阶段”
- `PersonaSpec` 解决“这个阶段像怎样的一个人格”
- `ArenaMessage` 解决“这些人格说了什么”
- `SummaryRecord` 解决“这场会议最后沉淀了什么”

## 6. 当前后端接入情况

接口封装在：

- `entry/src/main/ets/service/PersonaApi.ets`

已接通的关键接口：

- `GET /api/presets`
- `GET /api/profiles/:profileId`
- `POST /api/timeline/parse`
- `POST /api/agents/build`
- `POST /api/arena/run`

已预留但前端尚未充分发挥的能力：

- `POST /api/arena/stream`

这对 UI 设计的意义是：

- 当前可以先按“静态结果式体验”设计
- 后续完全可以平滑升级为“实时会议演绎体验”

## 7. 当前视觉风格总结

从现有代码里的 `MintColors`、卡片背景、圆角、阴影和光斑来看，当前项目视觉风格关键词是：

- 清透
- 柔和
- 科技感但不硬核
- 情绪化、陪伴感
- 偏内容产品而不是工具后台

当前常见视觉特征：

- 浅灰白背景
- mint 绿色主色
- lavender 紫灰辅助色
- 超大圆角
- 半透明卡片
- 毛玻璃质感
- 大面积顶部 Hero 区
- 柔和动效和渐入动画

## 8. UI 设计时建议重点强化的方向

### 8.1 强化“时间人格”而不是普通聊天

这个项目最大的差异点不是聊天，而是“同一个人在不同阶段和自己开会”。

建议在 UI 中强化：

- 年龄 / 时间段
- 阶段情绪
- 阶段立场差异
- 人格之间的冲突与呼应

### 8.2 区分三类页面气质

建议页面气质分层：

- 人物相关页
  - 更偏叙事、档案感、人生切片感
- 会议相关页
  - 更偏多角色、舞台感、讨论感
- 纪要相关页
  - 更偏编辑部、文档沉淀、内容传播感

### 8.3 把“选人”做成关键体验点

当前从代码看，“选择哪几个阶段人格进入会议”是最关键决策。

这块值得重点设计：

- 清晰的选择态
- 不同人格的差异化卡面
- 参会阵容预览
- 最终会议阵容确认

### 8.4 纪要模块可以做成高完成度展示区

纪要页已经比较成熟，适合优先做精致：

- 视觉模板统一
- 分享卡片更像成品
- 收藏 / 标签 / 搜索更清晰
- 提升内容沉淀价值感

## 9. 当前 UI 设计切入建议

如果你下一步要开始设计 UI，建议按这个顺序推进：

### 方案 A：先做主链路

1. 首页工作台
2. 人物详情页
3. 参会人选择页
4. 会议聊天页

适合先把核心体验跑顺。

### 方案 B：先做高完成度展示

1. 首页工作台
2. 纪要列表页
3. 纪要详情页
4. 分享卡片风格统一

适合比赛展示、答辩演示、作品集呈现。

### 方案 C：先突出产品差异化

1. 人物详情页
2. 时间线节点卡
3. 阶段人格选择页
4. 会议页中的人格对位关系

适合把“时间人格”这个概念打透。

## 10. 我建议你优先设计的页面

如果目标是“最快做出一套有说服力的 UI”，我建议优先做这 4 个页面：

1. 首页 `Index.ets`
2. 人物详情页 `ProfileDetail.ets`
3. 参会人选择页 `MeetingParticipantPage.ets`
4. 会议聊天页 `MeetingChatPage.ets`

原因：

- 它们串起了核心玩法
- 最能体现“时间人格”差异化
- 改完后产品气质会提升最明显

如果目标是“视觉成品感”，就优先做：

1. 首页 `Index.ets`
2. 纪要列表页 `SummaryListPage.ets`
3. 纪要详情页 `SummaryDetailPage.ets`

## 11. 技术实现上的 UI 注意点

基于当前代码，后续设计时要考虑这些实现边界：

- ArkUI 已经大量使用卡片式 `Column / Row / Stack` 组合
- 页面广泛使用 safe area 处理
- 已有较多过渡动画和渐入动效
- 颜色、圆角、间距、字体都有统一 style 文件

对应目录：

- `entry/src/main/ets/common/styles/Colors.ets`
- `entry/src/main/ets/common/styles/MintColors.ets`
- `entry/src/main/ets/common/styles/Spacing.ets`
- `entry/src/main/ets/common/styles/Typography.ets`
- `entry/src/main/ets/common/styles/BorderRadius.ets`
- `entry/src/main/ets/common/styles/Animation.ets`

也就是说，后续 UI 最好基于现有 design token 扩展，而不是完全推翻重来。

## 12. 结论

当前项目已经具备完整的产品骨架，不需要从 0 定义功能。

你后续做 UI 的核心任务，不是补功能说明，而是把已有产品结构重新“设计表达”出来，重点抓住三件事：

- 时间人格的差异感
- 会议过程的戏剧张力
- 纪要沉淀的内容价值
